# Phase 42: Entity Subject Field

## Status: COMPLETED

Add a human-readable `subject` (mnemonic name) to Requests, Plans, and Reviews so users can quickly identify what each entity is about in lists, detail views, and cross-references.

## Executive Summary

**Problem:**
When listing requests, plans, or reviews, the only identifying information is a trace ID (UUID). Users must read the full body or open the detail view to understand what a given entity is about. This makes the TUI tables, CLI lists, and linked navigation difficult to scan.

**Solution:**
Introduce a `subject` string field across all three entity types:

1. **Requests**: Set via `--subject "..."` CLI option, or generated by the agent when processing the request, with a text-extraction fallback.
2. **Plans**: Inherited from the parent request's subject (propagated at plan creation time).
3. **Reviews**: Inherited from the parent plan's subject (propagated at review creation time).

The subject is stored in each entity's frontmatter/metadata, displayed in CLI `list`/`show` output, and available for the future TUI redesign.

## Detailed Design

### 1. Subject Field Specification

| Property | Value |
|---|---|
| Field name | `subject` |
| Type | `string` (optional) |
| Max length | 80 characters (truncated with `…` if longer) |
| Default | Agent-generated from request body; text-extraction fallback |
| Allowed chars | Any printable UTF-8; no newlines |
| Persistence | Stored in YAML frontmatter of each entity's markdown file |

### 2. Subject Generation Strategy

Subject is determined by a **two-tier** approach:

#### Tier 1: Agent-Generated Subject (Primary)

When the agent processes a request and produces a plan response, the response schema **requires** a `subject` field. The agent is instructed to generate a concise, human-readable summary of the request's intent (like a commit message subject line).

- The agent prompt includes an instruction: *"Provide a `subject` field: a concise one-line summary (≤ 80 chars) describing the request's goal, suitable for display in lists and tables."*
- The `subject` field is added to the plan response schema / structured output format.
- On receiving the agent's response, the system **validates** the subject:
  - Non-empty string
  - Single line (no `\n`)
  - Length ≤ 80 characters
  - Not a repetition of the trace ID or generic filler (e.g., not just "Request")
- If validation passes → use the agent's subject.
- If validation fails → fall through to Tier 2.

#### Tier 2: Text-Extraction Fallback

Used when: (a) `--subject` is not provided, (b) agent omits subject, or (c) agent returns invalid subject.

1. Take the `description` argument (inline) or the first non-empty line of the file (with `--file`).
2. Strip leading markdown markers (`#`, `-`, `*`, `>`, numbers followed by `.`).
3. Strip leading/trailing whitespace.
4. If the result is longer than 60 characters, truncate at the last word boundary before 60 chars and append `…`.
5. Store the result as `subject`.

```typescript
export function extractFallbackSubject(text: string, maxLength = 60): string {
  const firstLine = text.split("\n").find((l) => l.trim().length > 0) || "";
  const cleaned = firstLine.replace(/^[\s#\-*>\d.]+/, "").trim();
  if (cleaned.length <= maxLength) return cleaned;
  const truncated = cleaned.substring(0, maxLength).replace(/\s+\S*$/, "");
  return truncated + "…";
}
```

#### Resolution Order

```
1. --subject "..." provided by user  →  use as-is (validate length)
2. Agent response includes subject   →  validate, use if valid
3. Fallback text extraction           →  extractFallbackSubject(description)
```

### 3. Subject Propagation Chain

```
Request (subject) ──→ Plan (subject) ──→ Review (subject)
        │                    │                    │
   set by user,         inherited from       inherited from
   agent-generated,     parent request       parent plan
   or fallback
```

- The plan creation code (`RequestProcessor.createPlanFromResponse`) copies the request's `subject` into the plan's frontmatter.
- The review creation code (`ReviewCommands`/review generator) copies the plan's `subject` into the review's frontmatter.
- If a parent entity has no subject (backward compatibility), the child entity's subject remains `undefined`.

### Step 2. Subject Generation & Validation

**Objective**: Create utilities for fallback subject extraction and agent-response subject validation.

**Files to create:**

- `src/helpers/subject_generator.ts` — Exports:
  - `extractFallbackSubject(text: string, maxLength?: number): string` — Text-extraction fallback.
  - `validateSubject(subject: unknown): string | null` — Validates agent-returned subject (non-empty, single-line, ≤ 80 chars). Returns cleaned subject or `null` if invalid.
  - `resolveSubject(options: { explicit?: string; agentSubject?: string; description: string }): string` — Resolution-order orchestrator: explicit → agent → fallback.

**Success Criteria:**

- `extractFallbackSubject` handles: single-line, multi-line, markdown-prefixed, empty, and very long inputs.
- `validateSubject` rejects: empty strings, multi-line strings, strings > 80 chars, generic filler.
- `resolveSubject` follows the three-tier resolution order.

**Planned Tests:**

- `tests/helpers/subject_generator_test.ts`:
  - `extractFallbackSubject`: short input, long input (truncated + `…`), markdown prefixes stripped, empty → `""`, multi-line → first line.
  - `validateSubject`: valid subject → returned, empty → `null`, multi-line → `null`, too long → `null`, generic filler → `null`.
  - `resolveSubject`: explicit wins over agent, agent wins over fallback, invalid agent falls through to fallback.

### Step 4. Agent Response Schema & Subject Propagation

**Objective**: Require `subject` in agent plan responses; propagate subject through the entity chain.

**Files to modify:**

- Agent prompt template (e.g., `Blueprints/Fragments/` or system prompt assembly) — Add instruction: *"Include a `subject` field in your response: a concise one-line summary (≤ 80 chars) of the request's goal."*
- Plan response schema / structured output definition — Add `subject: string` (required) to the expected response shape.
- `src/services/request_processor.ts` — When processing the agent's response:
  1. Extract `subject` from agent response.
  2. Validate via `validateSubject()`.
  3. If valid → update the request's frontmatter `subject` (upgrade from fallback) and copy into plan frontmatter.
  4. If invalid → keep the request's existing subject (user-provided or fallback) and copy into plan.
- `src/services/structured_plan_parser.ts` — Add `subject?: string` to `IStructuredPlanFrontmatter`.
- `src/cli/commands/review_commands.ts` (or review creation code) — When creating a review, copy plan's `subject` into review metadata.

**Subject Flow:**

```
1. User creates request:
   subject = --subject "..." || extractFallbackSubject(description)

2. Agent processes request, returns plan with subject:
   agentSubject = validateSubject(response.subject)
   if (agentSubject) {
     request.subject = agentSubject   // upgrade request subject
     plan.subject = agentSubject      // set plan subject
   } else {
     plan.subject = request.subject   // keep existing
   }

3. Review created from plan:
   review.subject = plan.subject
```

**Success Criteria:**

- Agent prompt includes subject instruction.
- Valid agent subject upgrades both request and plan subjects.
- Invalid/missing agent subject falls back gracefully to existing request subject.
- A request with `subject: "Dark Mode"` produces a plan with `subject: "Dark Mode"`.
- Review inherits plan's subject.
- Entities without subjects (pre-existing data) handled gracefully (`undefined`).

**Planned Tests:**

- `tests/services/request_processor_test.ts`:

  - Agent returns invalid/missing subject → request keeps fallback, plan inherits it.
  - Plan without request subject → plan subject is `undefined`.
- `tests/cli/review_commands_test.ts` (or equivalent):

## Backward Compatibility

- The `subject` field is **optional** in all schemas (`.optional()` in Zod, `?` in interfaces).
- Existing request/plan/review files without a `subject` field will continue to load and function normally.
- `PlanFrontmatterSchema` already uses `.passthrough()`, so unknown fields are preserved.
- No migration is needed. Old entities simply display no subject in the UI.

---

## Out of Scope

- **TUI integration**: The new TUI redesign (Phase TUI) will consume the `subject` field natively. No changes to the current TUI code are included in this phase.
- **Subject editing**: Editing/renaming a subject after creation is not included. This can be added later.
- **Subject search/filter**: Searching or filtering by subject in CLI or TUI is not included.
